# 知识库项目 Review 流程

## 执行摘要

| 项目 | 内容 |
|------|------|
| **触发词** | "review"、"审查"、"代码检查"、"验收"、"plan 审查" |
| **硬关卡** | Critical 问题必须修复后才能继续/关闭 |
| **核心输出** | 分级问题列表（Critical/Major/Minor/Info） |
| **禁止行为** | 只审查代码不读 spec、安全审查流于形式 |
| **下一步** | 当场修复 → 运行验证 → 继续 |

> 核心原则：**契约对齐、分层审查、问题分级、当场修复。**

---

## 一、Review 类型速查

| 用户意图                                    | 对应审查类型   | 触发条件                     |
|---------------------------------------------|----------------|------------------------------|
| "看看代码有没有问题" / "走查一下"           | 代码审查       | 编码完成后                   |
| "设计是否合理" / "架构有没有问题"           | 方案合理性审查 | 编码前或方案评审时           |
| "是否符合 spec" / "对照 behavior-spec 检查" | Spec 对齐审查  | 编码后或联调前               |
| "TDD 合规吗" / "测试覆盖够吗"               | TDD 合规审查   | 编码后或审查时               |
| "安全有没有问题" / "OWASP" / "漏洞"         | 安全审查       | 任何涉及认证/输入/输出的变更 |
| "可以关闭了吗" / "验收一下" / "最终检查"    | 关闭前验收     | issue 标记 in-review 时      |
| "plan 有没有问题" / "计划审查"              | Plan 审查      | 计划编写完成后、执行前       |

## 二、通用审查流程

所有审查类型遵循以下统一流程：

```
1. 读取上下文 → 2. 执行审查 → 3. 分级输出 → 4. 当场修复
```

### 步骤 1：读取上下文

**代码审查：**
- `docs/issues/{dir}/issue.md` — 了解验收标准
- `docs/issues/{dir}/specs/` — 功能/行为/API 规格
- `docs/issues/{dir}/plan.md` — 执行计划（了解预期改动范围）
- **测试代码**：`tests/{layer}/{name}.spec.ts` — 检查测试覆盖
- 实际代码 diff（`git diff` 或指定文件）

**设计/架构审查：**
- `docs/issues/{dir}/specs/feature-spec.md` — 功能边界
- `docs/adrs/` — 相关架构决策
- 设计稿 / 架构图（如有）

**安全审查：**
- `docs/issues/q-01-security-baseline/issue.md` — 安全基线
- 涉及认证/授权/输入验证的代码
- 新增的外部依赖（检查已知漏洞）

**Plan 审查：**
- `docs/issues/{dir}/specs/` — 三层规格
- `docs/issues/{dir}/issue.md` — 验收标准
- `docs/guide/workflow.md` — 流程规范

### 步骤 2：执行审查

根据审查类型执行具体检查：

- **代码审查** → 读取 `references/code-review-checklist.md`
- **安全审查** → 读取 `references/security-review-checklist.md`
- **TDD 合规** → 读取 `references/tdd-compliance-checklist.md`
- **架构合规** → 检查是否违反 ADR 决策（验证方案、响应格式、依赖引入）
- **Plan 审查** → 检查完整性、可执行性、TDD 合规性、依赖关系
- **Spec 对齐** → 逐条检查验收标准与实现对齐
- **关闭前验收** → 检查代码验收 + 测试验收 + Spec 验收 + 联调整合

### 步骤 3：分级输出

所有问题按严重级别分类：

| 级别     | 标识 | 含义                                          | 处理方式             |
|----------|------|-----------------------------------------------|----------------------|
| Critical | 🔴   | 阻塞性问题：功能错误、安全漏洞、spec 严重偏离 | 必须修复后才能继续   |
| Major    | 🟠   | 重要问题：代码质量差、性能隐患、边界未处理    | 建议修复，可讨论豁免 |
| Minor    | 🟡   | 轻微问题：命名不佳、冗余代码、格式问题        | 可选修复             |
| Info     | 🔵   | 建议：优化思路、最佳实践、未来改进方向        | 仅供参考             |

**输出格式：**

```markdown
## 审查摘要

- **审查类型**：{代码审查 / 设计审查 / 安全审查 / 验收}
- **审查对象**：{文件 / issue / plan}
- **总体结论**：{通过 / 有条件通过 / 不通过}
- **问题统计**：Critical 0 | Major 0 | Minor 0 | Info 0

---

## 发现的问题

### 🔴 Critical

1. **{问题简述}**
   - 位置：`{文件路径:行号}`
   - 详情：{具体描述}
   - 建议：{如何修复}
   - 依据：{spec/安全基线/最佳实践}

### 🟠 Major

（同上格式）

### 🟡 Minor

（同上格式）

### 🔵 Info

（同上格式）

---

## Spec 对齐检查

| 验收标准 | 状态 | 证据        |
|----------|------|-------------|
| {标准 1} | ✅/❌  | {文件:行号} |

---

## 修复确认

（如用户修复后复查，在此记录验证结果）
```

### 步骤 4：当场修复

审查发现问题后，**当场修复**，不生成持久化报告。

**修复流程：**
```
输出审查结果（对话中）
  ↓
用户确认或提出异议
  ↓
当场修复（用户或 Agent）
  ↓
运行验证命令确认修复
  ↓
继续下一步
```

## 三、子代理驱动模式下的审查派发

当使用 `subagent-driven-development` 执行开发时，审查通过子代理派发。

**强制审查时机：**
- 子代理驱动开发中每个任务后
- 完成主要功能后
- 合并到 main 前

**审查顺序（强制）：**
1. **Spec 对齐审查** — 代码是否符合 spec？
2. **代码质量审查** — 实现是否优雅？

**代码质量审查在 spec 对齐通过前禁止开始。**

**审查循环：**
reviewer 发现问题 → implementer 修复 → reviewer **重新审查** → 通过或再修复

**禁止跳过重新审查。**

## 四、接收审查反馈的原则

**核心原则：技术严谨优于社交舒适。**

**禁止的回应：**
- "你说得对！"（表演式认同）
- "好点子！" / " excellent feedback!"（过度社交化）
- "我这就改"（未经验证就实现）

**正确的回应：**
- 用技术语言重述需求
- 提出澄清问题
- 直接开始工作（行动 > 言语）

**处理流程：**
```
收到审查反馈时：
1. 完整阅读，不立即反应
2. 用自己的话重述要求（或提问澄清）
3. 对照代码库验证技术正确性
4. 评估：对本项目技术可行？
5. 回应：技术确认或有理有据的反驳
6. 实现：一次一项，每项测试
```

**模糊反馈处理：**
- **停止** —— 不要开始实现
- **询问** —— 澄清不清楚的项

**YAGNI 检查：**
审查者建议"完善地实现"某功能时：
1. grep 代码库确认实际使用情况
2. 未使用 → "此功能未被调用。删除它（YAGNI）？"
3. 已使用 → 按规范实现

**何时反驳：**
- 建议破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 对本技术栈不正确
- 存在兼容性/遗留原因
- 与架构决策冲突

**修复顺序：**
1. 先澄清所有不清楚的项
2. 按此顺序实现：
   - 阻塞性问题（功能破坏、安全）
   - 简单修复（拼写、导入）
   - 复杂修复（重构、逻辑）
3. 每项单独测试
4. 验证无回归

## 五、常见陷阱

| 陷阱                | 后果                      | 正确做法                             |
|---------------------|---------------------------|--------------------------------------|
| 只审查代码不读 spec | 遗漏交互状态或错误码      | 始终先读 spec 再审查                 |
| 忽略边界条件        | 线上出现空状态/错误态崩溃 | 对照 behavior-spec 检查所有状态      |
| 安全审查流于形式    | 漏洞上线                  | 对照 q-01-security-baseline 逐项检查 |
| 发现问题不分级      | 阻塞非阻塞问题混为一谈    | 使用 Critical/Major/Minor/Info 四级  |
| 验收不检查联调      | 前后端接口不匹配          | 必须端到端验证后再关闭               |
| **不检查测试代码**  | TDD 流于形式              | 必须检查 `.spec.ts` 存在性和覆盖度   |
| **不验证测试先行**  | 事后补测试                | 抽查 git log 确认测试先于实现        |
| **不检查架构合规**  | 违反 ADR 决策未被发现      | 必须检查验证方案、响应格式、依赖引入是否符合 ADR |
