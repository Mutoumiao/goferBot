### 5.3 后端模块启动依赖顺序

NestJS 模块注册顺序隐含依赖：

`ConfigModule`(全局) → `EventEmitterModule` → `ThrottlerModule` → `CommonModule` → `HealthModule`(独立) → `CacheModule` → `UserModule` → `AuthModule`(依赖 User+Role) → `QueueModule`(依赖 Redis) → `StorageModule`(依赖 MinIO) → `KnowledgeBaseModule`(依赖 Storage) → `SessionModule` → `ChatModule`(依赖 Session + RAG + ProviderRegistry) → `CompanionModule`(依赖 LangGraph + ProviderRegistry) → `SettingsModule` → `AdminModule`(依赖 Auth + User + PermissionSeeder 启动初始化) → `RagModule`

**关键变更**（v22）：
- `AuthModule` 新增对 Role 表的依赖，JwtStrategy 路径感知双 Cookie 解析
- `AdminModule` 含 `PermissionSeeder`（OnModuleInit 幂等初始化角色+权限）和 `SuperAdminBootstrap`（OnApplicationBootstrap 创建超管）
- `ChatModule` / `CompanionModule` / RAG 不再各自构造 LLM 客户端，统一通过 `ProviderRegistry` 获取
