#### JWT 双密钥架构

Access Token 和 Refresh Token 使用**独立密钥**签发，防止跨类型签名攻击。JWT payload 含 `app` 字段标识签发来源。

| Token | 密钥环境变量 | 默认有效期 | Web Cookie | Admin Cookie | Cookie maxAge |
|-------|------------|-----------|-----------|-------------|---------------|
| Access | `JWT_SECRET` | 2h (`JWT_EXPIRES_IN`) | `goferbot_web_access_token` | `goferbot_admin_access_token` | 15min |
| Refresh | `JWT_REFRESH_SECRET` | 7d (`JWT_REFRESH_EXPIRES_IN`) | `goferbot_web_refresh_token` | `goferbot_admin_refresh_token` | 7d |

**Cookie 隔离目的**：web 和 admin 共用同一域名时，旧版 Cookie 同名导致互相覆盖。App 级命名消除冲突，登出一端不影响另一端。

#### 认证端点路径重构

登录/注册/刷新/登出全部按 app 分路径：

| 功能 | Web 端点 | Admin 端点 |
|------|---------|-----------|
| 登录 | `POST /auth/web/login` | `POST /auth/admin/login` |
| 注册 | `POST /auth/web/register`（需邀请码） | - |
| 刷新 | `POST /auth/web/refresh` | `POST /auth/admin/refresh` |
| 登出 | `POST /auth/web/logout` | `POST /auth/admin/logout` |
| 当前用户 | `GET /auth/me`（通用，X-App-Context 决定返回角色） |
| RSA 公钥 | `GET /auth/public-key`（公开） |

**Legacy 端点已删除**：`/auth/login`、`/auth/refresh`、`/auth/register`（无 app 前缀）返回 404。
**模块拆分**：个人资料/头像从 Auth 迁至 User 模块——`GET/PATCH /user/me`、`POST /user/avatar`。

#### 邀请码注册机制

Web 注册必须提供邀请码，校验顺序：
1. `TEST_INVITATION_CODES` 环境变量万能码（逗号分隔，不入库，admin 不可见）
2. 数据库 `InvitationCode` 表（GF-XXXX-XXXX 格式，排除易混字符 0/O/1/I/L）

- **standard 码**：一码一账户（`usedBy` 唯一约束），使用后事务内原子关联
- **multi 码**：支持 `maxUses` 限制，仅增 `usedCount` 不设 `usedBy`
- **环境变量测试码**：`invitationCodeId` 设 null，前端显示"测试邀请码"标签
- **错误码**：`INVITATION_CODE_INVALID`、`INVITATION_CODE_USED`、`INVITATION_CODE_EXPIRED`、`INVITATION_CODE_MAX_USES`

#### Token Rotation 与并发安全

```
登录: createSessionWithTokenPair(app, 事务: AuthSession + RefreshToken)
         → sign AccessToken(payload含app) + RefreshToken → Cookie 设置（路径感知命名）

刷新: verify RefreshToken → findRefreshTokenByJtiHash
         → 三重检查: usedAt? / revokedAt? / session.revokedAt?
         → 生成新 jti → insertRefreshToken(新) 
         → markRefreshTokenUsed(旧) → sign 新 AccessToken + RefreshToken（保留原 app）
```

**并发保护** — 原子 SQL 防止 Token 竞态重放：
```sql
UPDATE "RefreshToken"
SET "usedAt" = now(), "replacedByTokenId" = ?
WHERE "jtiHash" = ? AND "usedAt" IS NULL AND "revokedAt" IS NULL
RETURNING "id"
```
只有一个并发请求能成功（RETURNING 返回行），其余触发 `TOKEN_REPLAY` → 整个 Session 被撤销。

#### jti Hash 安全机制

每次 RefreshToken 生成 UUID jti → SHA256 hash → 存入 `RefreshToken.jtiHash`。JWT payload 含明文 jti，数据库存 hash。即使数据库泄露也无法从 jtiHash 反推 JWT payload 中的 jti，不可能构造有效的 RefreshToken。

#### Token 链式追踪（审计/溯源）

RefreshToken 表两个外键形成完整旋转链：

| 字段 | 含义 |
|------|------|
| `parentTokenId` | 上一个 Token 的 ID |
| `replacedByTokenId` | 替代者 Token 的 ID |

用于安全审计追踪 Token 流转历史和攻击溯源。

#### Auth Redis — 独立连接 + Fail-Closed

AuthRedisService 使用**独立 Redis 连接**，不共用 Queue 或 Cache 的 Redis：

| 功能 | Redis Key 前缀 | TTL | 说明 |
|------|---------------|-----|------|
| Token 黑名单 | `token:blacklist:` | 与 token 剩余有效期一致 | 登出/撤销时写入 |
| 用户缓存 | `auth:user:` | 300s | 避免每次请求查 Prisma |
| 权限缓存 | `auth:permission:` | 300s | 避免每次查 RolePermission 表 |
| lastSeen 节流 | `auth:lastSeen:throttle:` | 60s (SETNX) | 防止高频更新 seelastSeen |

**Fail-Closed 策略**: 生产环境 Redis 不可用时，`isTokenBlacklisted()` 返回 `true`（拒绝所有请求）。开发环境降级允许通过。

#### Cookie 安全策略

| httpOnly | true | true |

#### CAPTCHA 完整实现

- **生成**: 4位随机字符（排除 0/O/1/I 混淆字符）→ SVG（4条贝塞尔干扰线 + 20个噪点）→ sharp → PNG buffer
- **存储**: Redis `captcha:{id}` + `captcha:img:{id}`，TTL=120s
- **验证**: 一次性消费 — 无论正确与否立即删除，防止暴力探测
- **集成**: 登录接口强制要求验证码
- **E2E 跳过**：`CAPTCHA_WHITELIST_ORIGINS` 环境变量，允许白名单 Origin 绕过验证码（仅非生产环境生效）
