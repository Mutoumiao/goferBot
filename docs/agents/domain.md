# 领域文档

工程技能在探索代码库时应如何消费本仓库的领域文档。

## 探索前，先阅读这些

- **`CONTEXT.md`** 位于仓库根目录，或
- 如果存在 **`CONTEXT-MAP.md`** 位于仓库根目录 —— 它指向每个上下文的 `CONTEXT.md`。阅读与主题相关的每一个。
- **`docs/adr/`** —— 阅读涉及你即将工作领域的 ADR。在多上下文仓库中，同时检查 `src/<context>/docs/adr/` 中上下文范围的决策。
- **`PROGRESS.md`** —— 每个上下文可能在其根目录有一个。如果存在，在编写代码前阅读它，了解进行中的内容、已完成的内容和下一步计划。
- **`CONSTRAINTS.md`** —— 仓库根目录。如果存在，在任何实现工作前阅读。包含以 MUST/MUST NOT 语言书写的不可协商规则。
- **`ARCHITECTURE.md`** —— 每个服务/模块可能在其根目录有一个。如果存在，在修改前阅读，了解该服务的职责、公共接口和依赖关系。

如果上述任何文件不存在，**静默继续**。不要标记其缺失；不要建议预先创建它们。生产者技能（`/grill-with-docs`）会在术语或决策真正被解决时惰性创建它们。

## 文件结构

单上下文仓库（大多数仓库）：

```
/
├── CONTEXT.md
├── CONSTRAINTS.md
├── PROGRESS.md
├── ARCHITECTURE.md
├── docs/adr/
│   ├── 0001-event-sourced-orders.md
│   └── 0002-postgres-for-write-model.md
└── src/
```

## 使用词汇表的术语

当你的输出命名一个领域概念（在问题标题、重构提案、假设、测试名称中）时，请使用 `CONTEXT.md` 中定义的术语。不要偏离词汇表明确避免的同义词。

如果你需要的概念尚未在词汇表中，这是一个信号 —— 要么你正在发明项目不使用的语言（请重新考虑），要么确实存在一个缺口（记录下来供 `/grill-with-docs` 处理）。

## 标记 ADR 冲突

如果你的输出与现有 ADR 矛盾，请明确提出来，而不是静默覆盖：

> _与 ADR-0007（event-sourced orders）矛盾 —— 但值得重新讨论，因为…_
