# ADR 0004: 知识库数据库与物理目录启动同步策略

## 状态

Accepted

## 背景

知识库采用本地存储方案，数据存储在用户目录下的 `knowledge-base/` 中：
- `docs/`：各知识库的物理目录（一级子目录即为一个知识库）
- `sidecar.db`：SQLite 数据库，其中 `knowledge_bases` 表记录知识库元信息

在应用出错或异常退出的场景下，可能出现以下不一致：
- **物理有、数据库无**：`docs/` 下已创建目录，但数据库 `knowledge_bases` 表没有对应记录。后续用户创建同名知识库时，因物理目录已存在而触发不可预期的错误（如后续 `renameSync` 冲突或路径漂移）
- **数据库有、物理无**：数据库中存在未删除的知识库记录，但对应的物理目录已丢失（用户手动删除、磁盘错误等）。前端会显示一个无法访问的知识库

当前系统以数据库为隐式权威来源，但缺乏启动时的对齐机制，导致不一致状态持续累积。

## 决策

采用**启动扫描同步策略**：sidecar 启动并完成数据库初始化后，立即扫描 `docs/` 物理目录，与 `knowledge_bases` 表进行双向对齐。

### 同步规则

| 场景 | 处理行为 |
|---|---|
| 物理目录存在，数据库有同名未删除记录且 `path` 一致 | 无需处理 |
| 物理目录存在，数据库有同名未删除记录但 `path` 不一致 | 更新数据库 `path` 为实际物理路径（修复路径漂移） |
| 物理目录存在，数据库无同名未删除记录 | 自动创建数据库记录（生成新 ID，`name` = 目录名，`path` = 完整路径） |
| 物理目录存在，数据库仅有同名**已删除**（回收站）记录 | 忽略，由回收站恢复逻辑单独处理同名冲突 |
| 数据库有未删除记录，但 `path` 对应的物理目录不存在 | 移入回收站（`deleted_at` = 当前时间戳，若 `.trash/` 存在则尝试移动） |

### 扫描范围

- 只扫描 `docs/` 下的一级子目录（递归子目录属于知识库内部文件结构，不视为独立知识库）
- 忽略 `.trash/` 及其他以 `.` 开头的隐藏目录
- 目录名为空或非法的跳过（由操作系统保证）

### 触发时机

sidecar 启动时，数据库初始化（含迁移）完成后立即执行一次全量同步。此后正常运行期间不再自动扫描，依赖各 API 的独立文件操作保证一致性。

## 后果

### 正面

- **自愈能力**：应用重启后自动修复"物理有、数据库无"的残留状态，避免创建同名知识库失败
- **数据一致性**：数据库记录始终与物理目录对齐，减少前端显示"幽灵知识库"（无法打开）的情况
- **无用户感知**：同步在后台完成，用户无需手动触发"重建"或"修复"

### 负面

- **启动延迟**：扫描和同步在启动时串行执行，如果 `docs/` 下知识库数量极多，会增加 sidecar ready 的时间（本地文件系统扫描通常在毫秒级，可忽略）
- **意外导入**：用户手动在 `docs/` 下放了一个文件夹，重启应用后会被自动识别为知识库。这在本地存储方案下是可接受的行为
- **误删风险**："数据库有、物理无"的知识库会被自动移入回收站。如果用户因磁盘挂载问题暂时无法访问目录，重启应用可能导致知识库被误删。考虑到这是桌面本地应用，磁盘挂载场景极罕见，风险可控

## 替代方案

### 方案 B：懒同步 / 手动修复

不一致时不自动处理，只在用户创建知识库冲突时返回特定错误码，由用户手动删除残留目录或选择"修复"。

**拒绝原因**：增加用户认知负担。本地桌面应用应尽可能自动处理异常状态，而非要求用户理解物理目录与数据库的关系。

### 方案 C：数据库唯一权威，忽略物理目录

启动时不扫描物理目录。创建知识库时如果物理目录已存在，直接复用该目录。

**拒绝原因**：复用目录会引入未知状态（目录内可能已有文件），且无法解决"数据库有、物理无"的幽灵知识库问题。

## 备注

- 本同步策略与 ADR-0003 的"RAG 索引同步"是正交的：本 ADR 处理 `knowledge_bases` 表与物理目录的对齐，ADR-0003 处理文件内容变更后 `document_chunks` / `vec_document_chunks` / `fts_document_chunks` 的同步
- 同步时创建的新知识库记录不触发 RAG 自动索引（无文件变更），避免启动时产生大量 Embedding 调用。后续 #04 实现索引队列后，可在同步完成后将新增知识库标记为"待索引"
