# ADR 0002: RAG 向量存储使用 SQLite + sqlite-vec + 混合搜索

## 状态

已接受 (Accepted)

## 上下文

知识库应用需要实现 RAG（检索增强生成）功能，核心包括：

1. **索引流程**：用户导入文档 → LangChain 文本分割 → 调用 Embedding API 获取向量 → 存储向量
2. **检索流程**：用户提问 → Embedding query → 向量相似度搜索 → Top-K 文档块拼入 Prompt → LLM 生成回答

向量存储方案的选择直接影响：
- 系统依赖复杂度（是否需要额外服务或原生扩展）
- 检索性能（数据量增大后的查询延迟）
- 检索质量（纯向量语义搜索可能遗漏关键词精确匹配）
- 包体积和部署复杂度
- 后续迁移成本

## 决策

RAG 向量存储采用 **SQLite + `sqlite-vec` 扩展 + 混合搜索（向量相似度 + 全文搜索/关键词匹配）**。

具体实现：
- `document_chunks` 表的 `embedding` 字段仍为 BLOB 类型，存储浮点数组
- 使用 `sqlite-vec` SQLite 扩展创建 `vec0` 虚拟表，建立 HNSW 向量索引
- 使用 SQLite FTS5 扩展创建全文索引表，对 `content` 和 `file_path` 建立全文索引
- **混合搜索流程**：
  1. 向量搜索：通过 `sqlite-vec` 的 `vec_distance_cosine` 获取语义相似的 Top-K 文档块（O(log N)）
  2. 全文搜索：通过 FTS5 对 query 做关键词匹配，获取关键词命中的 Top-K 文档块
  3. 融合排序：使用 RRF（Reciprocal Rank Fusion）或加权评分合并两种搜索结果，去重后取最终 Top-K

## 权衡

### 优势

1. **高性能向量检索**：`sqlite-vec` 提供 HNSW 索引，检索时间复杂度 O(log N)，单知识库数万到数十万文档块仍可保持毫秒级延迟。
2. **混合搜索提升召回率**：纯向量搜索对同义词、语义相关的内容效果好，但可能对精确关键词（如专有名词、型号、代码函数名）不够敏感。结合 FTS5 全文搜索，可以同时覆盖语义相似和关键词精确匹配。
3. **单数据库方案**：向量和全文索引都在同一个 SQLite 文件中，不需要额外的服务进程或数据库，备份和迁移只需复制一个 `.db` 文件。
4. **跨平台支持成熟**：`sqlite-vec` 提供预编译的各平台二进制扩展（Windows `.dll`、macOS `.dylib`、Linux `.so`），Tauri 打包时可将扩展文件作为 sidecar 资源一并分发。

### 劣势

1. **原生扩展依赖**：`sqlite-vec` 和 FTS5 都是原生扩展（FTS5 内置于大多数 SQLite 发行版，但 `sqlite-vec` 需要额外加载）。需要处理跨平台的扩展加载路径。
2. **包体积增加**：`sqlite-vec` 各平台的预编译二进制约 1-5MB/平台，会增加 Tauri 应用的打包体积。
3. **索引构建成本**：HNSW 索引在首次导入或重建时需要计算和写入索引结构，比纯 BLOB 插入慢。大批量导入时索引构建可能成为瓶颈。
4. **维护复杂度**：需要同时维护向量索引（`vec0` 虚拟表）和全文索引（FTS5 虚拟表），索引同步（文档块增删改时两个索引都要更新）增加代码复杂度。

### 被拒绝的替代方案

- **SQLite BLOB + 应用层余弦相似度**：拒绝原因：O(N) 全表扫描，数据量增大后检索延迟线性增长，内存占用高，无法支撑长期演进。
- **LanceDB**：拒绝原因：引入新的存储引擎，与现有 SQLite 数据存储分裂，增加架构复杂度。
- **ChromaDB**：拒绝原因：需要单独的服务进程或 Python 运行时，与 Node.js sidecar 架构不匹配。
- **纯向量搜索（无全文）**：拒绝原因：对精确关键词匹配和专有名词的召回率不足，混合搜索能显著提升检索质量。

## 后果

1. **需要同时维护两套索引**：文档块变更时，`document_chunks` 表、`vec0` 向量虚拟表、`fts_document_chunks` 全文虚拟表三者需要同步更新。sidecar 的索引逻辑需要封装统一的"写入/更新/删除"操作，确保数据一致性。
2. **SQLite 连接初始化变复杂**：启动时需要加载 `sqlite-vec` 扩展（`db.loadExtension('/path/to/vec0')`），扩展文件路径在不同平台不同，需要 Tauri 在运行时根据平台提供正确的路径。
3. **需要定义混合搜索的融合策略**：第一版采用 RRF（Reciprocal Rank Fusion）作为默认融合算法，公式：`score = Σ(1 / (k + rank))`，其中 k 通常取 60。后续可扩展为支持加权融合（向量权重 vs 全文权重可调）。
4. **Embedding API 调用不变**：索引和检索仍然调用外部 Embedding API 获取向量，`sqlite-vec` 只改变存储和检索方式，不改变 Embedding 的来源。
5. **预留性能调优空间**：HNSW 索引参数（`M`、`efConstruction`、`efSearch`）可根据知识库规模动态调整。
