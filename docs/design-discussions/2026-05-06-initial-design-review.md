## 问题 1：为什么要引入 Hono Sidecar？

### 原始设计

原始设计概要的三层架构：
- Tauri Rust：窗口管理、启动/监控 sidecar、提供 appData 路径
- Hono Sidecar：全部业务（LLM 问答、RAG、LangChain、文件 I/O、SQLite、Embedding）
- 前端 Vue：UI 渲染、状态管理、直接 HTTP 调用 sidecar

### 挑战

提出三个核心质疑：

1. **Tauri Rust 本身可以直接做文件 I/O 和 SQLite**，引入 sidecar 意味着多进程、多端口、多监控逻辑的额外复杂度。这个复杂度是为了什么买单？
2. **LangChain 是 Node.js 生态的库**，这是引入 sidecar 的主要动机吗？有没有更轻量的替代？例如：前端直连 LLM API、Rust 原生实现 RAG（candle/ort）、Tauri IPC 直接暴露命令。
3. **Sidecar 的端口冲突和生命周期管理**（固定 11451，冲突递增，Rust 监控重启）引入了新的故障模式。如果 sidecar 启动失败，用户体验是什么？

### 用户回应

用户明确动机：
- Tauri 只做轻量桌面壳（窗口管理、文件系统 I/O）
- 业务 LLM 相关全部使用 Node 完成，更好接入 LangChain 生态
- 用户是 Node 开发工程师，更容易掌握架构与代码能力
- **要减少 Rust 的使用**

### 最终决策

**确认引入 Node.js Hono Sidecar。**

理由：Node 生态优先 + 团队技能栈匹配。这是难以逆转的架构决策，后续写入 ADR-0001。

## 问题 2：数据持久化策略为什么分裂？

### 原始设计

原始设计中存在两种持久化方式：

| 数据 | 存储方式 |
|------|----------|
| 会话 & 消息 | SQLite |
| 文档块 & 向量 | SQLite |
| 知识库元信息 | SQLite |
| **问答历史** | **JSON (`qa-history.json`)** |
| **用户配置** | **JSON (`config.json`)** |

PRD 中甚至写"第一版先用 JSON，后续迁移 SQLite"。

### 挑战

提出三个问题：

1. PRD 写"第一版先用 JSON，后续迁移 SQLite"——但 sidecar 已经有了完整的 SQLite 连接。为什么历史记录不直接进 `sidecar.db` 的 `sessions`/`messages` 表？额外维护 JSON 增加了序列化逻辑，而且"迁移"本身就是高风险操作。
2. 配置用 JSON 是合理的（人类可读、用户可能手动编辑），但历史记录用 JSON 的动机是什么？
3. 如果历史用 JSON，那 `sessions` 表的作用是什么？PRD 的 SQLite Schema 里已经有 `sessions` 和 `messages` 表。这是两套真相来源（source of truth），谁为主？

### 用户回应

用户确认：**结构化数据统一用 SQLite 管理，只有配置才使用 JSON。**

### 最终决策

- **结构化数据**（会话、消息、知识库、文档块、向量）→ **SQLite**
- **用户配置**（LLM provider、API Key、温度等）→ **JSON** (`config.json`)
- 消除 `qa-history.json`，不再存在 JSON/SQLite 双轨制

## 问题 3：知识库是"物理目录"还是"虚拟分组"？

### 原始设计

知识库管理页设计：
- 左侧：一级知识库列表，支持新建
- 右侧：资源管理器视图，显示文件/文件夹，双击进入子文件夹

PRD 数据目录：`knowledge-base/docs/`（按知识库分子目录）

### 挑战

提出四个核心问题：

1. "新建知识库"创建的是什么？是只在 SQLite 里插入一条记录，还是在 `docs/` 下真的创建一个物理子目录？
2. 如果知识库对应物理目录，删除 = 删除整个目录。知识库删除的边界是什么？
3. 如果知识库只是虚拟分组，所有文件扁平放在 `docs/` 下，右侧的"子文件夹"怎么实现？
4. RAG 索引的范围是什么？设计概要说"Node sidecar 直接读取 knowledge-base/docs/"——这是全局读取，不是按知识库。

### 用户回应

用户明确：
- **新建知识库就是一个物理文件夹**，便于后续针对这个知识库进行文件搜索
- 删除机制：第一次删除先不彻底删除，有 30 天的回收站，可以随时恢复
- 除非在回收站彻底删除

用户最初认为"暂时有 30 天的回收站"，但后续在问题 3.1 中调整。

### 最终决策

**知识库 = 物理子目录。**

`docs/<知识库名>/` 下的所有文件/子目录即该知识库的内容。RAG 索引按知识库粒度进行（`POST /knowledge-bases/:id/index` 只扫描该子目录）。

## 问题 3.1：回收站的实现机制

### 挑战

用户提出了 30 天回收站的概念，需要明确实现方式。给出三种方案：

1. **物理移动**：删除时把 `docs/知识库A/` 移动到 `docs/.trash/知识库A-<timestamp>/`，恢复时移回原位，30 天后 sidecar 定时任务扫描删除。
2. **数据库标记**：文件不动，SQLite 加 `deleted_at` 字段，30 天后物理删除。
3. **混合**：数据库标记 + 物理移动。

并追问：
- 用哪种机制？
- 恢复时名称冲突怎么处理？
- 30 天清理是 sidecar 启动时扫描一次，还是需要常驻定时器？

### 用户回应

- 采用**方案 1（物理移动）**
- 恢复时同名冲突采用**重命名后面加上"-副本"**
- 30 天清理可以忽略，**只需要一个回收站即可，不需要删除**（第一版不实现自动清理）

### 最终决策

- 回收站 = 物理移动至 `.trash/<知识库名>-<timestamp>/`
- 恢复时冲突重命名为 `<原名>-副本`
- **第一版不实现 30 天自动清理**

## 问题 4：RAG 向量存储——SQLite BLOB + 应用层余弦相似度能撑多久？

### 原始设计

- Embedding 向量存入 SQLite `document_chunks.embedding` 字段（BLOB）
- 检索时"SQLite 向量相似度搜索（余弦相似度）"

### 挑战

提出三个问题：

1. **SQLite 没有原生向量索引**。几万条记录时每次检索都要全表扫描加载所有向量到内存做余弦相似度，时间复杂度 O(N)。
2. **向量维度问题**。OpenAI `text-embedding-3-small` 是 1536 维，一条记录 = 6KB。1 万条 = 60MB，10 万条 = 600MB。每次检索全量加载到 Node.js 内存。
3. **为什么不用专门的向量数据库？** 如 `sqlite-vec`（SQLite 扩展）、`lancedb`、`chromadb`，都支持 HNSW 索引（O(log N)）。

### 用户回应（后续已变更，见最终决策）

- 最初接受 SQLite BLOB + 应用层余弦相似度（简单、零依赖）
- 后续明确改为 **SQLite + sqlite-vec + 混合搜索**

### 最终决策

- **向量存储：SQLite + `sqlite-vec` 扩展 + 混合搜索（向量相似度 + FTS5 全文搜索）**
- `sqlite-vec` 提供 HNSW 向量索引，检索复杂度 O(log N)
- 同时使用 FTS5 对 `content` 和 `file_path` 建立全文索引
- **混合搜索流程**：
  1. 向量搜索：通过 `sqlite-vec` 获取语义相似 Top-K
  2. 全文搜索：通过 FTS5 获取关键词匹配 Top-K
  3. 融合排序：RRF（Reciprocal Rank Fusion）合并两种结果，去重后取最终 Top-K
- `document_chunks` 表仍需保留 `embedding` BLOB 字段作为原始数据备份
- 需要同时维护向量索引（`vec0` 虚拟表）和全文索引（FTS5 虚拟表）

## 问题 5：前端如何发现 sidecar 的地址？

### 原始设计

- sidecar 端口（11451 或递增后的实际端口）写入 `knowledge-base/.sidecar-port`
- 前端直接 `fetch(http://127.0.0.1:PORT)`

### 挑战

提出三个问题：

1. 前端怎么知道 PORT 是多少？给出三种方案：
   - A：Rust 通过 `invoke` 暴露端口给前端
   - B：前端自己通过 `readTextFile` 读取 `.sidecar-port`
   - C：固定端口 11451，前端 hardcode
2. 如果 sidecar 崩溃重启并分配了新端口，前端持有的旧端口怎么办？需要端口变更通知机制。
3. Tauri 的 CSP 安全策略：动态端口需要在 `tauri.conf.json` 中允许 `localhost:*`。

### 用户回应

**按照方案 A，Rust 还是尽量简洁。**

### 最终决策

- **方案 A**：Rust 启动 sidecar 后，通过 `invoke('get_sidecar_port')` 把端口给前端
- sidecar 崩溃重启后，Rust 通过 Tauri event 向前端广播新端口
- 前端 HTTP 请求统一封装 `sidecarClient`，内部处理端口变更和重试
- CSP 中允许 `http://localhost:*`

## 问题 6：文件导入的完整链路是什么？

### 挑战

Tauri v2 中前端通过 `dialog` 选择文件后获得临时访问权限，但 sidecar 是独立进程，**没有**这个权限。给出三种方案：

**方案 A：前端搬运**
- 前端 `open()` 选文件 → `readBinaryFile()` 读取 → HTTP POST 到 sidecar → sidecar 保存
- 缺点：大文件会被前端加载到内存再转 Base64

**方案 B：Rust 中转**
- 前端调用 Rust IPC → Rust 打开对话框、读取文件、直接写入目标目录 → 前端刷新列表
- 缺点：Rust 直接做文件保存 = 打破"Rust 只做轻量壳"的分层

**方案 C：Rust 直传 sidecar**
- 前端调用 Rust IPC → Rust 打开对话框、读取文件 → Rust HTTP POST 到 sidecar → sidecar 保存
- 最符合分层，但 Rust 需要实现 HTTP client

并追问：
1. 文件导入走哪个方案？
2. 大文件（后续 PDF）怎么考虑？
3. 导入的目标子目录由谁决定、怎么传递？

### 用户回应

**方案 C，Rust 不用过分简洁，还是可以尽量利用 Rust 做一些高性能操作。**

### 最终决策

- **方案 C（Rust 直传 sidecar）**
- Rust 负责打开对话框、读取文件、HTTP POST 到 sidecar
- sidecar 保存到 `knowledge-base/docs/<知识库>/<子目录>/`
- 这实际上微调了"Rust 只做轻量壳"的边界：Rust = 窗口管理 + 高性能文件 I/O + sidecar 生命周期管理

## 问题 7：标签栏的"首页"到底是什么？

### 原始设计

- 问答会话：可多开
- 单例页面（知识库管理、设置、对话历史）：只能开一个
- "首页"标签始终保留、无法关闭
- 最右侧 `+` 按钮新建问答会话

### 挑战

提出四个问题：

1. "首页"是一个有 ID 的会话，还是一个特殊的占位符？
2. 如果用户在"首页"已经聊了几轮，然后点左侧边栏的"消息图标"，会发生什么？（切换到首页 vs 新建空标签）
3. 如果消息图标 = 切换到首页，那新建问答的唯一入口就是 `+`？
4. 如果消息图标 = 新建会话，那"首页"就不再有"始终保留"的特殊性。

给出了两种方案：

**复杂方案（建议）**："首页"不是持久化会话，而是快捷入口。一旦用户发送第一条消息，"首页"自动变成有 ID 的真实会话，同时系统创建新的首页占位符。

**简单方案**："首页"就是一个普通会话（ID 固定如 `default`），始终保留。消息图标只是切换到它。`+` 才是新建入口。

### 用户回应

**不用简单，维持按照你的建议，"首页"不是一个持久化会话，而是一个新建会话的快捷入口。它始终存在。一旦用户发送了第一条消息，"首页"自动变成一个有 ID 的真实会话。**

### 最终决策

- **首页 = 始终保留的占位符，不是一个持久化会话**
- 用户在首页输入第一条消息后，首页自动升格为真实会话（存入 SQLite `sessions` 表）
- 系统同时创建一个新的首页占位符标签
- 消息图标行为：切换到当前首页占位符（如果没有则新建）
- `+` 按钮：新建独立的新会话标签

## 问题 8：知识库索引什么时候触发？谁来触发？

### 原始设计

RAG 索引流程：sidecar 读取文件 → LangChain 分块 → Embedding API → 存入 SQLite。但没有明确触发时机。

### 挑战

给出四种方案：

| 方案 | 触发时机 | 优点 | 缺点 |
|------|----------|------|------|
| A 手动 | 用户点击"重建索引" | 可控、省 API 调用 | 用户会忘 |
| B 自动即时 | 每导入一个文件立即索引 | 立即可检索 | 批量导入 = 多次 API 调用，慢且贵 |
| C 队列后台 | 导入后加入队列，sidecar 后台逐个处理 | 批量不卡 UI、自动完成 | 需要队列状态管理 |
| D 文件监听 | `fs.watch` 监听 docs 目录 | 完全自动化 | 跨平台不可靠、批量变更触发大量事件 |

并追问：
1. 索引触发机制选哪个？
2. 文件级别的索引状态是否需要展示给用户？
3. 问答页是否需要感知知识库的索引状态？

### 用户回应

**方案 C 队列后台方案，上传文件后则可以排序索引，然后给出索引进度条即可。**

### 最终决策

- **方案 C：队列后台索引**
- 文件导入后加入索引队列，sidecar 按顺序逐个处理
- 前端显示索引进度条
- 知识库管理页的文件列表显示索引状态（已索引/排队中/失败）
- 支持手动触发"重建索引"

## 问题 9：LLM 配置是"单提供商全局配置"还是"多提供商保存 + 快速切换"？

### 原始设计

设置页配置一个当前激活的 LLM 提供商（OpenAI / Claude / DeepSeek / 自定义 / Ollama），保存到 `config.json`。配置模型只有一个 `llmProvider` 字段。

### 挑战

提出三个问题：

1. 用户可能有多个 API Key（OpenAI + DeepSeek）。每次切换都要去设置页重新填 Key 和模型，体验糟糕。
2. 很多 AI 客户端的做法：设置页保存所有提供商配置，对话页顶部提供下拉框快速切换当前会话使用的模型。
3. 如果第一版只做单提供商全局配置，后续加"每会话模型切换"需要重构 `config.json` 结构和 `POST /chat` API。

给出了建议的 `config.json` 多提供商结构：

```json
{
  "providers": {
    "openai": { "apiKey": "...", "model": "gpt-4o", "baseUrl": "" },
    "deepseek": { "apiKey": "...", "model": "deepseek-chat", "baseUrl": "" },
    "custom": { "apiKey": "...", "model": "...", "baseUrl": "..." },
    "ollama": { "enabled": true, "url": "...", "model": "..." }
  },
  "embeddingProvider": { "provider": "openai", "apiKey": "...", "model": "..." },
  "temperature": 0.7,
  "defaultChatProvider": "openai"
}
```

并追问：第一版是否需要在对话页提供模型快速切换？

### 用户回应

**没问题，按照你的建议。**

### 最终决策

- **多提供商配置**：`config.json` 保存所有提供商的 Key、模型、Base URL
- **每会话模型切换**：对话页顶部下拉框快速切换，切换仅影响当前会话
- **会话表加字段**：`provider` + `model`，记录该会话创建时的配置快照
- 设置页是全局配置的入口，对话页是快速切换的入口

## 问题 10：Session 与标签页是 1:1 还是 1:N？

### 原始设计

"点击恢复 session 到新标签"。

### 挑战

提出两种方案：

**方案 A：1:1（推荐）**
- 一个 session 同一时间只能存在于一个标签
- 恢复时：若未打开则新建标签；若已打开则激活已有标签
- 关闭标签 ≠ 删除 session

**方案 B：1:N**
- 一个 session 可在多个标签打开
- 恢复时总是新建标签
- 缺点：跨标签状态同步复杂

并追问边界情况：如果当前标签是空首页占位符，用户从历史恢复已有 session，应该替换首页还是新建标签？

建议：**1:1 + 恢复时优先复用首页占位符**

### 用户回应

**方案 1，一个对话始终保持一个标签页。**

### 最终决策

- **Session : Tab = 1:1**
- 恢复历史会话时，若当前激活标签是**空的首页占位符**，直接用该 session 替换首页占位符内容
- 否则新建标签加载 session
- 若该 session 已在其他标签打开，切换到那个标签

## 问题 11：知识库检索开关是什么粒度？

### 原始设计

问答对话页顶部有一个"知识库检索开关"。

### 挑战

提出三种粒度：

| 粒度 | 行为 | 存储位置 |
|------|------|----------|
| 全局 | 开关一开，所有会话的所有消息都检索 | `config.json` |
| 会话级 | 每个会话独立决定是否启用 RAG | `sessions.use_knowledge_base` |
| 消息级 | 每条消息单独决定是否检索 | 不持久化 |

并追问：
1. 检索开关的粒度？
2. 未配置 Embedding API 时的降级行为？

### 用户回应

**不需要添加这个，默认首页对话只需要手动输入 @ 引入想要咨询的知识库来对话。否则就是正常对话。后续将会在知识库页面对应单个知识库可以支持对话操作。**

### 最终决策（重大变更）

**取消全局/会话级的"知识库检索开关"，改为 `@提及` 机制。**

具体规则：
- 用户在输入框输入 `@` 弹出知识库下拉选择列表（模糊搜索）
- 选择后渲染为蓝色 pill/tag，支持多选，可删除
- `@提及` 的知识库只影响**当前这条消息**的 RAG 上下文
- 不 `@任何知识库` = 普通对话，不做 RAG

## 问题 13：知识库管理页的搜索和排序范围是什么？

### 原始设计

顶部工具栏：面包屑导航、搜索、排序、添加文件。

### 挑战

追问：

2. 排序维度：名称、日期、类型、大小？
3. 文件列表 API 是层级返回（仅当前目录）还是递归扁平返回？

**搜索只要文件名即可，排序按照简单排序就行，搜索返回的文件我希望是显示搜索的所有文件。搜索出来内容如果是文件夹就正常进入对应目录与面包屑。面包屑左边提供回退与前进操作。回退时将回到搜索状态。**

## 问题 14：Sidecar 启动时序和前端"就绪"状态

### 挑战

追问启动流程的完整链路：

1. 前端怎么知道 sidecar 已经 ready？轮询 `/health`？Rust event 通知？
2. 如果 sidecar 启动失败，前端显示什么？
3. 窗口应该等 sidecar ready 后再显示，还是先显示再显示 Loading？
4. sidecar 崩溃重启后，前端如何感知？
5. Loading 期间左侧边栏和顶部标签栏是否显示？

给出了完整建议：

1. **启动时序**：Rust 启动 sidecar → sidecar 启动成功后写入 `.sidecar-port` → Rust 通过 Tauri event `sidecar-ready` 通知前端（携带端口）→ 前端收到后开始正常调用
2. **窗口显示**：Tauri 窗口先显示，但前端显示全局 Loading 遮罩（"正在启动服务..."），直到收到 `sidecar-ready` 或超时（30 秒）
3. **超时/失败**：30 秒内未收到 ready，前端显示错误页："服务启动失败，请检查日志或重启应用"，提供"重试"按钮
4. **崩溃重启**：Rust 检测到 sidecar 退出 → 自动重启（递增端口）→ 通过 `sidecar-restarted` event 通知前端携带新端口 → 前端 `sidecarClient` 更新端口，自动重试失败请求
5. **Loading 期间**：显示 Splash，不渲染主 UI

### 用户回应

**这里全部按照你的建议来。**

### 最终决策

- **启动时序**：Rust 启动 sidecar → sidecar ready → Rust emit `sidecar-ready` (含端口) → 前端解除 Loading
- **Loading 状态**：窗口先显示，前端全屏 Loading 遮罩（"正在启动服务..."）
- **超时处理**：30 秒超时 → 错误页 + "重试"按钮（调用 Rust IPC 重新启动 sidecar）
- **崩溃恢复**：Rust 自动重启 sidecar + emit `sidecar-restarted` (含新端口) → 前端 `sidecarClient` 自动更新
- **Splash 期间**：不渲染主 UI（侧边栏、标签栏均隐藏）
