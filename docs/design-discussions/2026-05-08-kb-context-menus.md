## 问题 1：新功能作为新增 issue 还是修改 #03？

### 挑战

#03 状态已为 closed（已完成），但验收标准中完全没有右键菜单和文件操作。是重新打开 #03，还是新增独立 issue？

### 选项

- **A**：重新打开 #03，将右键菜单操作加入其验收标准
- **B**：#03 保持关闭，新增独立 issue（#03b）专门补齐右键菜单和文件操作

### 最终决策

**选项 B。** 重新打开已关闭的 issue 会混淆 git 历史和已完成的工作量。新增 issue 更清晰，而且右键菜单涉及的交互深度足以支撑一个独立 issue。

## 问题 2：知识库"置顶"的持久化方式

### 挑战

置顶状态是保存在前端本地，还是持久化到 SQLite？

### 选项

- **A（前端本地）**：保存在 `localStorage` 或 `config.json` 中。简单但换设备/清缓存后丢失。
- **B（SQLite 持久化）**：`knowledge_bases` 表增加 `is_pinned`（布尔）和 `sort_order`（整数）字段。跨会话一致。

### 最终决策

**选项 B（SQLite）。** 置顶是知识库管理的核心属性，与"修改名称/图标"同属元信息层，理应持久化。`sort_order` 字段也为未来拖拽排序预留空间。

## 问题 3：知识库"修改图标"的图标来源

### 挑战

图标来源有三种可能：MDI 预设库、Emoji、上传自定义图片。

### 选项

- **A（MDI 预设库）**：只从 Material Design Icons 中选择，前端弹出图标网格选择器
- **B（支持 Emoji）**：除 MDI 外还允许 Emoji
- **C（上传自定义图片）**：允许上传 PNG/SVG

### 最终决策

**选项 A（MDI 预设库），默认图标 `mdi-database`。** 与现有图标体系一致，无需引入额外依赖。`icon` 字段存储 MDI 名称（如 `"mdi-database"`），前端直接渲染为 `<span class="mdi mdi-database" />`。

## 问题 4：文件删除是物理删除还是移入回收站？

### 挑战

PRD 中知识库级别的删除定义为"移入 `.trash/`"，但文件/文件夹级别的删除没有定义。

### 选项

- **A（物理删除）**：直接删除文件，不可恢复。简单直接，配合二次确认弹窗 mitigate 误删风险。
- **B（知识库内回收站）**：每个知识库内部维护 `.trash/` 子目录。恢复时移回原位。
- **C（系统回收站）**：调用系统 API 移入系统回收站。最符合直觉但 Tauri v2 支持有限。

### 最终决策

**选项 A（物理删除）+ 前端二次确认弹窗。** 知识库回收站的设计初衷是保护"整个知识库"不被误删（涉及大量文件和 RAG 索引）。单个文件删除的恢复成本较低（重新导入即可）。引入文件级回收站会大幅增加复杂度，远超第一版范围。

## 问题 5：文件"移动到"和"复制到"的目标范围

### 挑战

目标选择范围：仅限当前知识库内，还是可以跨知识库？

### 选项

- **A（仅限当前知识库内）**：目标选择器只展示当前知识库的目录树。实现简单，不涉及索引状态同步。
- **B（可跨知识库）**：目标选择器展示所有知识库及其目录树。文件从一个知识库物理迁移到另一个。

### 最终决策

**选项 B（可跨知识库）。** 用户明确需要跨知识库移动/复制。UI 为弹窗形式——左栏选知识库，右栏选目标目录，顶部面包屑导航。底部按钮"移动至此"/"复制至此"基于当前面包屑位置直接确认。

---

## 问题 6：命名冲突的统一处理规则

### 挑战

新建文件夹、重命名、复制到同一目录都会遇到命名冲突。

### 选项

- **A（自动重命名）**：冲突时自动加序号
- **B（阻止并提示）**：冲突时弹窗提示用户手动修改
- **混合策略**：新建文件夹用 A，重命名用 B，复制用 A

### 最终决策

**混合策略调整为：新建文件夹默认名 `未命名文件夹_短时间戳`（如 `未命名文件夹_20250508`），立即聚焦在重命名输入框中让用户修改。** 时间戳极大降低冲突概率，且直接让用户参与命名。重命名冲突时阻止并提示。复制冲突时自动加 `(1)` 后缀。

## 问题 7：跨知识库移动/复制后，RAG 索引如何同步？

### 挑战

跨库操作涉及物理文件移动 + 数据库 `document_chunks` 更新 + RAG 索引同步。

### 选项

- **A（即时同步）**：移动/复制完成后，sidecar 立即从源知识库删除索引记录，将文件加入目标知识库的索引队列，后台重新处理
- **B（懒同步）**：只操作物理文件，不立即更新索引，文件显示为"未索引"
- **C（禁止跨库移动已索引文件）**：若文件已索引，弹窗提示用户可能丢失索引

### 最终决策

**选项 A（即时同步），但前提是跨库操作 API 内部复用 #04 的索引队列机制。** 用户操作文件后期望索引状态自动保持一致。本 issue 先实现物理文件操作 + 前端 UI，索引同步标记为依赖 #04 的后续补充项（由 #04b 负责）。

## 问题 8：文件操作的 Sidecar API 路由设计

### 挑战

需要补充新建文件夹、重命名、移动、复制的 API，且跨库操作的路径参数存在歧义。

### 选项

- **风格 A（资源导向）**：`POST /knowledge-bases/:id/folders`、`PATCH /knowledge-bases/:id/files/:path`
- **风格 B（动作导向）**：`POST /knowledge-bases/:id/files/:path/rename`

### 最终决策

**混合风格：**
- 知识库内操作（新建文件夹、重命名）用资源导向：`POST /knowledge-bases/:id/folders`、`PATCH /knowledge-bases/:id/files/:path`
- 跨库操作（move/copy）用动作导向 + 独立端点：`POST /files/move`、`POST /files/copy`，body 中显式携带 `sourceKbId` / `targetKbId`，避免路径参数歧义

## 问题 9：右键菜单是自定义 Vue 组件还是原生浏览器 contextmenu？

### 选项

- **A（自定义 Vue 组件）**：右键时阻止默认菜单，渲染自定义浮动面板。样式可控、支持图标、与 Tailwind 一致。
- **B（原生 contextmenu）**：使用 Tauri 原生菜单 API。样式不可控、与 Vue 状态集成复杂。

### 最终决策

**选项 A（自定义 Vue 组件）。** 与 Tailwind 设计系统一致，支持 MDI 图标，易于与 Pinia store 集成。

---

## 问题 10：知识库"修改资料"的编辑形式

### 选项

- **A（弹窗表单）**：弹出模态框，包含名称输入框 + 图标选择网格 + 保存/取消
- **B（行内编辑）**：列表项本身变成可编辑状态
- **C（分离操作）**：右键菜单分成"重命名"和"更换图标"两个独立项

### 最终决策

**选项 A（弹窗表单）。** 知识库列表左侧边栏宽度有限，行内编辑空间不足。"修改资料"在语义上是配置操作，弹窗更符合用户心智模型。名称和图标在同一弹窗内可以同时修改。

---

## 问题 11：文件重命名是行内编辑还是弹窗输入？

### 选项

- **A（行内编辑）**：文件名区域变为可编辑输入框，Enter 保存，Esc 取消
- **B（弹窗输入）**：弹出小模态框输入新名称

### 最终决策

**选项 A（行内编辑）。** 与 Windows/macOS 资源管理器标准行为一致，用户直觉匹配。焦点移开自动保存。只修改文件名，锁定扩展名。

---

## 问题 12~13：移动/复制弹窗的交互设计

### 最终决策

弹窗布局：
- 顶部：面包屑导航（含回退/前进按钮）+ 动态标题（显示源文件名）
- 左侧：知识库列表（选择目标知识库）
- 右侧：仅显示文件夹列表目录（点击后进入对应目录，面包屑同步更新）
- 底部："取消" + "移动至此"/"复制至此"按钮

基于当前面包屑位置直接确认目标，无需二次确认。

## 问题 14：删除操作的二次确认弹窗差异化设计
