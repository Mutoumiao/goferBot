# ADR 0003: 文件操作与 RAG 索引同步策略

## 状态

Accepted

## 背景

#03（知识库管理）建立了知识库的物理目录映射和文件导入机制，#03b 补充了完整的文件操作功能（新建文件夹、重命名、跨库移动/复制）。#04（RAG 索引）使用 `sqlite-vec` + FTS5 对知识库文件建立向量索引和全文索引。

当用户执行文件操作时，物理文件系统状态会改变，但 RAG 索引数据（`document_chunks`、`vec_document_chunks`、`fts_document_chunks`）不会自动感知这些变化。如果不同步更新索引，会导致以下问题：

- **跨库移动**：文件从知识库 A 移到 B 后，A 的检索仍返回已不存在的文件，B 的检索找不到该文件
- **知识库/文件重命名**：路径变更后，索引中存储的 `file_path` 与实际路径不一致
- **删除**：已删除文件的索引记录变成脏数据

因此需要明确文件操作后如何保持索引一致性。

## 决策

采用**即时同步策略**：文件操作完成后，sidecar 立即将受影响的文件加入索引队列，由后台队列逐个处理索引更新。

具体规则：

| 文件操作 | 索引同步行为 |
|---|---|
| 跨库移动文件 | 从源知识库的三张索引表中移除该文件的所有记录；将文件加入目标知识库的索引队列，后台重新分块、Embedding、写入索引 |
| 跨库复制文件 | 将副本加入目标知识库的索引队列，后台重新建立索引 |
| 知识库重命名 | 同步更新 `document_chunks` 中所有该知识库记录的 `file_path` 前缀；同步更新 `vec_document_chunks` 和 `fts_document_chunks` 的关联记录 |
| 文件重命名 | 同步更新 `document_chunks` 中对应记录的 `file_path`；同步更新向量表和全文索引表 |
| 文件删除 | 从三张索引表中移除该文件的所有记录 |

索引队列复用 #04 的后台队列机制（`RecursiveCharacterTextSplitter` 分块 → Embedding API → 写入 `document_chunks` + `vec_document_chunks` + `fts_document_chunks`）。

## 后果

### 正面

- **用户体验一致**：用户操作文件后期望检索结果自动反映最新状态，无需手动触发"重建索引"
- **避免脏数据**：索引与物理文件系统保持同步，减少检索到不存在文件或遗漏新文件的情况
- **复用现有机制**：索引队列已在 #04 实现，文件操作只需将任务加入队列，不引入新的并发模型

### 负面

- **实现复杂度增加**：每个文件操作 API（move/copy/rename）都需要在事务中同时更新物理文件、数据库元信息和索引状态，出错时需要回滚
- **Embedding API 调用成本**：跨库移动/复制会触发目标知识库的重新索引，产生额外的 Embedding API 调用费用
- **延迟问题**：索引是后台队列处理，文件移动后不会立即可检索，需要等待队列处理完成。前端需显示索引进度以管理用户预期
- **#03b 与 #04 的耦合**：#03b 的实现必须预留索引同步的扩展点，实际索引同步逻辑由 #04b 在 #04 完成后补充

## 替代方案

### 方案 B：懒同步 / 标记脏数据

文件操作只更新物理文件和数据库元信息，不立即更新索引。被影响的文件或知识库标记为"索引过时"状态，用户手动触发"重建索引"时才更新。

**拒绝原因**：用户操作文件后期望检索结果自动保持一致。懒同步会导致"文件存在但搜不到"或"文件已删仍能搜到"的困惑状态，增加用户认知负担。

### 方案 C：禁止跨库移动已索引文件

如果文件已建立索引，跨库移动时弹窗阻止用户操作，要求用户先删除索引再移动。

**拒绝原因**：过度限制用户操作。跨库移动是合理的需求（如整理知识库分类），禁止会显著降低产品可用性。
