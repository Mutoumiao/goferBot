# 验证体系设计文档

> 建立统一的验证体系，确保 RAG SDK 在多模块架构下的工程质量。

---

## 1. 核心目标

- **类型安全**：确保代码的类型正确性
- **行为一致**：模块行为必须符合设计预期
- **独立自治**：每个功能模块均可独立测试
- **协同工作**：确保多模块之间的集成稳定性
- **防止回归**：修改现有代码后不破坏已有功能

---

## 2. 验证体系分层

系统由以下五个维度组成，构成完整的验证闭环：

1. **Type 验证**：基于 TypeScript 的静态类型检查
2. **Spec 验证**：基于 Zod 的运行时数据结构校验
3. **Unit Test**：基于 Vitest 的单元逻辑测试
4. **Demo 验证**：模块级别的最小可运行示例
5. **Integration**：跨模块协同工作的集成测试

---

## 3. 目录结构规范

在单包结构基础上，测试按模块组织：

```
packages/rag-sdk/
  ├── src/
  │   ├── types.ts              # 共享类型
  │   ├── schema.ts             # Zod Schema
  │   ├── interfaces.ts         # 接口定义
  │   ├── errors.ts             # 错误模型
  │   ├── pipeline.ts           # Pipeline 抽象
  │   ├── vector-store.ts       # 向量存储接口
  │   ├── indexing/             # 索引模块
  │   ├── runtime/              # 运行时模块
  │   └── observability/        # 可观测性模块
  ├── __tests__/
  │   ├── core/                 # core 契约测试（schema、errors、interfaces）
  │   ├── indexing/             # indexing 模块单元测试
  │   ├── runtime/              # runtime 模块单元测试
  │   ├── observability/        # observability 模块单元测试
  │   └── integration/          # 跨模块集成测试
  ├── demo/
  │   ├── indexing.ts           # 索引流水线 demo
  │   └── runtime.ts            # 检索流水线 demo
  └── package.json
```

---

## 4. 技术栈选型

验证体系必须且仅允许使用以下工具：

- **TypeScript**：负责全局类型检查
- **Vitest**：作为统一的测试运行框架
- **Zod**：负责 Schema 定义与数据清洗
- **tsx**：用于直接运行 TypeScript Demo 脚本

---

## 5. 各层验证执行要求

### 5.1 Type 验证（强制）

- **要求**：所有代码必须通过 `tsc` 编译检查
- **执行命令**：`pnpm type-check`（在 rag-sdk 包内）

### 5.2 Spec 验证（强制）

- **要求**：所有输入/输出接口必须定义 Zod Schema，并使用 `.parse()` 进行严格校验
- **范围**：外部输入（用户/API）、跨模块调用边界

### 5.3 Unit Test（强制）

- **要求**：每个模块必须包含 `__tests__/{module}/` 目录下的测试
- **核心逻辑覆盖率**：关键路径（chunk、embed、index、retrieve、postprocess）需达到基准要求

### 5.4 Demo 验证（强制）

- **要求**：每个模块必须提供 `demo/` 目录下的最小闭环示例
- **运行方式**：`npx tsx demo/indexing.ts`

### 5.5 集成测试

- **范围**：重点验证 `indexing + runtime`、`runtime + observability` 等关键链路
- **位置**：`__tests__/integration/`

---

## 6. 各模块最低验收标准

| 模块 | 验收项 |
|------|--------|
| **core** | Schema 测试（Zod parse）、类型检查、错误体系测试、接口兼容性测试 |
| **indexing** | Chunker 测试（空输入、层级信息）、Embedder 测试（Mock OpenAI、维度校验）、Indexer 测试（VectorRecord 映射）、Pipeline 测试（阶段状态） |
| **runtime** | Retriever 测试（Query 对象、kbId 过滤）、Postprocessor 测试（分数过滤、rerank、budget trim、trace 记录）、Pipeline 测试（阶段耗时） |
| **observability** | Tracer 测试（生命周期、多 Observer）、错误隔离测试 |

---

## 7. 工程自动化配置（包级 Scripts）

`packages/rag-sdk/package.json` 必须实现以下脚本：

```json
{
  "scripts": {
    "build": "tsc",
    "type-check": "tsc --noEmit",
    "test": "vitest run",
    "test:watch": "vitest",
    "demo:indexing": "tsx demo/indexing.ts",
    "demo:runtime": "tsx demo/runtime.ts"
  }
}
```

---

## 8. 开发流水线

每完成一个功能模块或重大特性修改，必须依次执行：

1. `pnpm type-check` — 类型检查
2. `pnpm test` — 单元测试
3. `pnpm demo:indexing` / `pnpm demo:runtime` — 本地 Demo 运行
4. `pnpm -r build` — 全局构建验证

---

## 9. 禁止事项

- 禁止缺失 Demo 或测试用例
- 严禁仅依赖手动运行来验证逻辑
- 禁止在 `runtime` 内部编写 BLEU/MRR 等评测指标（后续由 server 或独立 eval 模块负责）
- 禁止在测试中使用真实 API Key（必须使用 Mock）

---

## 10. 关键变更记录

| 变更项 | 原设计 | 新设计 | 原因 |
|--------|--------|--------|------|
| core 测试位置 | `__tests__/core/` | 保留 | 测试 schema、errors、interfaces |
| adapters 测试 | `__tests__/adapters/` | **删除** | adapters 模块已删除 |
| 集成测试范围 | `core + indexing` | `indexing + runtime`、`runtime + observability` | 覆盖实际业务链路 |
| Demo 运行 | `pnpm --filter @rag-sdk/indexing demo` | `npx tsx demo/indexing.ts` | 单包结构简化 |
