# RAG SDK 项目概述与架构定位

> 本文档基于 GoferBot 项目现有架构重新整理，将原独立多包设计方案适配为单包多模块结构。
> 最后更新：2026-05-27

---

## 1. 项目背景

GoferBot 是一个基于 Vue 3 + NestJS 的 AI Workspace 应用，RAG（检索增强生成）是其核心能力之一。RAG SDK 作为独立工具库，负责文档索引构建和语义检索，被后端服务（`@goferbot/server`）消费。

**当前状态**：接口层已冻结，实现层完全空白。详见 `../STATUS.md`。

---

## 2. 架构定位

### 2.1 在 GoferBot 中的位置

```
用户上传文件
    ↓
@goferbot/server — DocumentService.upload() — MinIO 存储 + PostgreSQL 记录
    ↓
BullMQ 队列 — Worker Handler（在 server 中实现）
    ↓
@goferbot/rag-sdk — 分块 / 向量化 / 索引写入（本 SDK 职责）
    ↓
PostgreSQL pgvector 扩展（chunks 表含 embedding 列）
```

```
用户提问
    ↓
@goferbot/server — ChatService.streamChat()
    ↓
@goferbot/rag-sdk — 语义检索（本 SDK 职责）
    ↓
pgvector 向量搜索 + PostgreSQL 查询内容
    ↓
LLM 生成回答
```

### 2.2 职责边界

**RAG SDK 负责：**
- 文档分块策略（Chunking）
- 文本向量化（Embedding）
- 向量索引写入（Indexing）
- 混合检索（向量检索 + 关键词检索 + RRF 融合）
- 检索后处理（Rerank、Budget Trim、Context Assembly）
- 索引/检索过程的可观测性（Trace、Metrics）

**RAG SDK 不负责：**
- 文件解析（PDF/Word 等，由 server 或后续 Parser 模块处理）
- 向量数据库底层操作（由 server 的 `IVectorStore` / `PgVectorStore` 处理）
- 关键词数据库底层操作（由 server 的 `IKeywordStore` / PostgreSQL FTS 处理）
- 业务编排（由 server 的 Worker Handler / ChatService 处理）
- LLM 生成（由 server 的 ChatService 处理）

### 2.3 一句话总结

> SDK 是**纯工具库**，负责 RAG 的核心算法和流程编排（分块、嵌入、索引、混合检索、后处理、观测），但不碰任何外部资源（文件系统、数据库、LLM API）。外部资源由 server 实现接口后注入 SDK。

### 2.4 边界示意图

```
用户上传文件
    ↓
server: DocumentService.upload() → MinIO + PostgreSQL
    ↓
server: BullMQ Worker
    - 解析文件 → 纯文本
    - 构造 DocumentSource
    - 调用 SDK: IChunker.chunk(doc) → chunks
    - 调用 SDK: IEmbedder.embed(texts) → vectors
    - 调用 SDK: IIndexer.index(chunks, vectors) → PostgreSQL pgvector（单事务）
    - 更新 PostgreSQL status
    ↓
用户提问
    ↓
server: ChatService.streamChat()
    - 构造 Query 对象（含 kbIds）
    - 调用 SDK: IRetriever.retrieve(query) → RetrievalCandidate[]
      - SDK 内部: 向量检索（pgvector）+ 关键词检索（PostgreSQL FTS）+ RRF 融合
    - 调用 SDK: Postprocessor.process(candidates) → selected
    - 调用 SDK: assembleContext(selected) → context 字符串
    - server: 拼接 system prompt + context + 用户问题
    - server: 调用 LLM API 生成回答
    ↓
用户看到回答
```

### 2.3 包引用方向

```
@goferbot/server → @goferbot/rag-sdk（单向依赖）
```

RAG SDK 作为纯工具库，**禁止反向依赖 server**。

## 3. 模块划分（单包多模块）

原独立多包方案（core / runtime / indexing / adapters / observability / eval / utils）适配为单包内的模块目录：

```
packages/rag-sdk/
├── src/
│   ├── index.ts              # 统一导出入口
│   ├── types.ts              # 共享数据模型（原 core/types）
│   ├── schema.ts             # Zod Schema 定义（原 core/spec）
│   ├── interfaces.ts         # 能力接口（原 core/interfaces）
│   ├── errors.ts             # 错误模型（原 core/errors）
│   ├── pipeline.ts           # Pipeline 类型抽象（原 core/pipeline）
│   ├── vector-store.ts       # 向量存储接口（新增，解耦 server）
│   ├── indexing/             # 离线索引构建（原 indexing 包）
│   │   ├── chunkers/         # 分块策略实现
│   │   ├── embedders/        # 向量化实现
│   │   ├── indexers/         # 索引写入实现
│   │   └── pipeline.ts       # 索引流水线编排
│   ├── runtime/              # 在线检索编排（原 runtime 包）
│   │   ├── retrievers/       # 检索实现
│   │   ├── postprocessors/   # 检索后处理
│   │   └── pipeline.ts       # 检索流水线编排
│   └── observability/        # 可观测性（原 observability 包，精简版）
│       ├── types.ts          # Trace / Metrics 类型
│       └── tracer.ts         # 默认 Tracer 实现
├── docs/                     # 设计文档
├── __tests__/                # 单元测试
├── demo/                     # 最小可运行示例
├── package.json
├── tsconfig.json
└── STATUS.md
```

### 3.1 模块职责对照表

| 新目录 | 原设计包 | 职责 |
|--------|----------|------|
| `src/types.ts` / `src/schema.ts` / `src/interfaces.ts` | `packages/core` | 共享领域契约：类型、Zod Schema、接口、错误、Pipeline 抽象 |
| `src/vector-store.ts` | — | 向量存储接口（从 server 解耦） |
| `src/indexing/` | `packages/indexing` | 离线数据构建：分块、向量化、索引写入、流水线编排 |
| `src/runtime/` | `packages/runtime` | 在线检索：检索、检索后处理、上下文组装、流水线编排 |
| `src/observability/` | `packages/observability` | 可观测性：Trace、Metrics、Observer 接口 |
| `__tests__/` | `packages/*/tests` | 单元测试（按模块组织） |
| `demo/` | `packages/*/demo` | 最小可运行示例 |

### 3.2 删除的模块

| 原模块 | 删除原因 |
|--------|----------|
| `adapters` | 当前仅 OpenAI 一种适配，直接实现在 `indexing/embedders/` 即可，无需独立适配层 |
| `utils` | 工具函数直接放在对应模块内，或作为独立文件放在 `src/` 根目录 |

## 4. 命名空间

统一使用 `@goferbot/rag-sdk` 命名空间：

```typescript

import type { IChunker, IEmbedder, IRetriever, IVectorStore } from '@goferbot/rag-sdk'
```

## 5. 关键设计决策

1. **扁平化 core**：将原 `core/` 目录的内容提升到 `src/` 根目录（types/schema/interfaces/errors/pipeline/vector-store），减少不必要的目录深度。
2. **删除 adapters**：当前仅 OpenAI 一种适配，直接实现在 `indexing/embedders/openai.ts`，避免职责重叠。
3. **向量存储接口内聚**：新增 `src/vector-store.ts`，定义 `IVectorStore` 接口，彻底解耦对 server 的反向依赖。
4. **Zod 强制**：所有外部输入（用户/API）和跨模块调用边界必须使用 Zod Schema 校验。
5. **错误链式追溯**：所有错误支持 `cause` 参数，便于调试底层故障。
6. **状态机驱动**：索引流程必须按 `uploaded → parsing → chunking → indexing → ready` 推进。
7. **可观测性内置**：Trace 和 Metrics 作为一等公民，每个 Pipeline 自动收集阶段耗时和中间结果。

---

## 6. 与原独立多包方案的主要差异

| 维度 | 原独立多包方案 | 当前单包多模块方案 |
|------|----------------|-------------------|
| 包数量 | 7 个独立包（core / runtime / indexing / adapters / observability / eval / utils） | 1 个包，接口扁平化 + 3 个模块目录（indexing / runtime / observability） |
| core 结构 | `packages/core/src/` | 扁平化到 `src/types.ts` / `src/schema.ts` / `src/interfaces.ts` |
| adapters | 独立的 `packages/adapters` | **删除**，OpenAI 实现直接放在 `indexing/embedders/` |
| 向量存储接口 | 依赖 server 的 `IVectorStore` | **内聚到 SDK**，定义在 `src/vector-store.ts` |
| 观测系统 | 独立的 `packages/observability` | 精简为 `src/observability/`，内置到 Pipeline |
| 评测系统 | 独立的 `packages/eval` | 暂不实现，后续按需扩展 |
| 检索策略 | 仅向量检索 | **混合检索（向量 + 关键词 + RRF）** | 主流方案，提升召回率 |
| 发布方式 | 每个包独立发布到 NPM | 作为 GoferBot 的 workspace 包统一构建 |
| 依赖管理 | 包之间通过 npm 版本依赖 | 模块之间通过相对路径引用 |
