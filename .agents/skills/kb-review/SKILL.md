# 知识库项目 Review 流程

> 本文档定义 GoferBot 项目的分层审查体系。
> 核心原则：**契约对齐、分层审查、问题分级、当场修复。**

---

## 一、Review 类型速查

| 用户意图                                    | 对应审查类型   | 触发条件                     |
|---------------------------------------------|----------------|------------------------------|
| "看看代码有没有问题" / "走查一下"           | 代码审查       | 编码完成后                   |
| "设计是否合理" / "架构有没有问题"           | 方案合理性审查 | 编码前或方案评审时           |
| "是否符合 spec" / "对照 behavior-spec 检查" | Spec 对齐审查  | 编码后或联调前               |
| "TDD 合规吗" / "测试覆盖够吗"               | TDD 合规审查   | 编码后或审查时               |
| "安全有没有问题" / "OWASP" / "漏洞"         | 安全审查       | 任何涉及认证/输入/输出的变更 |
| "可以关闭了吗" / "验收一下" / "最终检查"    | 关闭前验收     | issue 标记 in-review 时      |
| "plan 有没有问题" / "计划审查"              | Plan 审查      | 计划编写完成后、执行前       |

---

## 二、通用审查流程

所有审查类型遵循以下统一流程：

```
1. 读取上下文 → 2. 执行审查 → 3. 分级输出 → 4. 当场修复
```

### 步骤 1：读取上下文

审查前必须读取以下资料（按审查类型选择）：

**代码审查：**
- `docs/issues/{dir}/issue.md` — 了解验收标准
- `docs/issues/{dir}/specs/` — 功能/行为/API 规格
- `docs/issues/{dir}/plan.md` — 执行计划（了解预期改动范围）
- **测试代码**：`tests/issues/{dir}/*.spec.ts` — 检查测试覆盖
- 实际代码 diff（`git diff` 或指定文件）

**设计/架构审查：**
- `docs/issues/{dir}/specs/feature-spec.md` — 功能边界
- `docs/adrs/` — 相关架构决策
- 设计稿 / 架构图（如有）

**安全审查：**
- `docs/issues/q-01-security-baseline/issue.md` — 安全基线
- 涉及认证/授权/输入验证的代码
- 新增的外部依赖（检查已知漏洞）

**Plan 审查：**
- `docs/issues/{dir}/specs/` — 三层规格
- `docs/issues/{dir}/issue.md` — 验收标准
- `docs/guide/workflow.md` — 流程规范

### 步骤 2：执行审查

根据审查类型执行具体检查（详见下文各节）。

### 步骤 3：分级输出

所有问题按严重级别分类：

| 级别     | 标识 | 含义                                          | 处理方式             |
|----------|------|-----------------------------------------------|----------------------|
| Critical | 🔴   | 阻塞性问题：功能错误、安全漏洞、spec 严重偏离 | 必须修复后才能继续   |
| Major    | 🟠   | 重要问题：代码质量差、性能隐患、边界未处理    | 建议修复，可讨论豁免 |
| Minor    | 🟡   | 轻微问题：命名不佳、冗余代码、格式问题        | 可选修复             |
| Info     | 🔵   | 建议：优化思路、最佳实践、未来改进方向        | 仅供参考             |

**输出格式：**

```markdown
## 审查摘要

- **审查类型**：{代码审查 / 设计审查 / 安全审查 / 验收}
- **审查对象**：{文件 / issue / plan}
- **总体结论**：{通过 / 有条件通过 / 不通过}
- **问题统计**：Critical 0 | Major 0 | Minor 0 | Info 0

---

## 发现的问题

### 🔴 Critical

1. **{问题简述}**
   - 位置：`{文件路径:行号}`
   - 详情：{具体描述}
   - 建议：{如何修复}
   - 依据：{spec/安全基线/最佳实践}

### 🟠 Major

（同上格式）

### 🟡 Minor

（同上格式）

### 🔵 Info

（同上格式）

---

## Spec 对齐检查

| 验收标准 | 状态 | 证据        |
|----------|------|-------------|
| {标准 1} | ✅/❌  | {文件:行号} |

---

## 修复确认

（如用户修复后复查，在此记录验证结果）
```

### 步骤 4：当场修复

审查发现问题后，**当场修复**，不生成持久化报告。

**为什么不需要归档：**
- 审查结果直接体现在对话中，修复后立即验证
- 历史审查记录可通过 git log 追溯
- 减少文件维护负担，避免命名混乱

**修复流程：**
```
输出审查结果（对话中）
  ↓
用户确认或提出异议
  ↓
当场修复（用户或 Agent）
  ↓
运行验证命令确认修复
  ↓
继续下一步
```

## 三、代码审查（Code Review）

### 触发时机
- 编码完成后，提交前
- 用户说"帮我看看代码"、"走查一下"、"review 这段代码"

### 检查清单

#### 3.1 契约对齐
- [ ] 代码实现是否覆盖 issue 中的所有验收标准？
- [ ] 前端实现是否对齐 behavior-spec 中的交互状态（loading/empty/error/success/partial）？
- [ ] 后端 API 是否对齐 api-spec 中的端点、DTO、错误码？
- [ ] 数据模型是否对齐 spec 中的类型定义？

#### 3.2 代码质量
- [ ] **简洁优先**：是否用最小代码量解决问题？有无过度设计？
- [ ] **精准修改**：是否只修改了必要的文件？有无无关改动？
- [ ] **错误处理**：是否为不可能的情况添加了不必要的错误处理？（项目规范：不为不可能的情况加错误处理）
- [ ] **类型安全**：TypeScript 类型是否准确？有无 `any` 滥用？
- [ ] **命名**：变量/函数/类名是否清晰表达意图？

#### 3.3 前端专项
- [ ] Vue 组件是否遵循组合式 API 规范？
- [ ] Pinia Store 是否按功能模块拆分？
- [ ] UI 组件是否使用 shadcn-vue 标准组件？
- [ ] 颜色是否使用 Pencil tokens（`bg-surface-1`, `text-text-primary`）？
- [ ] Class 管理是否使用 `cn()` + `class-variance-authority`？
- [ ] 图标是否使用 `lucide-vue-next`？
- [ ] 响应式布局是否考虑移动端？

#### 3.4 后端专项
- [ ] NestJS 模块/控制器/服务分层是否清晰？
- [ ] API 端点是否遵循 RESTful 规范？
- [ ] DTO 是否使用 `nestjs-zod` 进行验证？
- [ ] 错误响应是否统一格式 `{ error: string }`，不暴露内部信息？
- [ ] 认证中间件是否正确注入 user/session？
- [ ] 数据库操作是否使用 Prisma 事务（需要时）？

#### 3.5 测试（TDD 合规检查）
- [ ] **新增功能是否有对应 `.spec.ts` 测试文件？**
- [ ] **测试是否覆盖正常路径和错误路径？**
- [ ] **测试是否使用真实数据库（项目规范：集成测试必须 hit 真实数据库，不 mock）？**
- [ ] **测试是否在实现代码之前编写？**（检查 git log 或询问开发者）
- [ ] **测试是否可运行且通过？**（要求提供 `npx vitest run` 输出）

**测试红线**：
- 无 `.spec.ts` 文件 → 🔴 Critical
- 测试只有 happy path → 🟠 Major
- 测试在实现之后补写 → 🟠 Major（首次警告，重复违规升级 Critical）

## 四、方案合理性审查（Plan & Architecture Review）

> 注意：本审查关注**方案的逻辑合理性和技术可行性**，不涉及视觉设计细节。
> 视觉设计审计（截图分析、AI Slop 检测、像素级检查）请使用 `/gstack-design-review`。

### 触发时机
- 编码前，方案设计完成后
- 用户说"这个方案合理吗"、"架构有没有问题"、"plan 审查"

### 检查清单

#### 4.1 功能边界
- [ ] 方案是否覆盖了 feature-spec 中的所有用户故事？
- [ ] 是否明确了范围外内容（防止范围蔓延）？

#### 4.2 交互状态完整性
- [ ] 是否定义了所有状态的视觉表现（loading/empty/error/success/partial）？
- [ ] 错误场景是否有恢复路径？
- [ ] 边界条件（空状态、无权限、网络断开）是否处理？

#### 4.3 技术可行性
- [ ] 方案是否超出团队技术栈能力？
- [ ] 新增依赖是否必要？有无更轻量替代方案？
- [ ] 性能影响是否可接受？

#### 4.4 一致性
- [ ] 是否与现有设计系统一致？
- [ ] 是否复用了现有组件/工具函数？
- [ ] 命名是否与项目惯例一致？

#### 4.5 TDD 合规性（Plan 审查新增）
- [ ] 每个任务是否以"编写失败测试"开始？
- [ ] 每个任务是否以"运行测试确认通过"结束？
- [ ] 测试文件路径是否明确声明？
- [ ] 测试是否覆盖 spec 中的所有交互状态？

## 五、安全审查（Security Review）

### 触发时机
- 涉及认证/授权/输入/输出的代码变更
- 新增外部依赖
- 用户说"安全有没有问题"、"OWASP"

### 检查清单

#### 5.1 认证与授权
- [ ] 密码是否使用 bcrypt 加密（≥10 rounds）？
- [ ] JWT 是否设置合理过期时间？
- [ ] 刷新令牌是否安全存储？
- [ ] 敏感端点是否有认证保护？
- [ ] 权限检查是否在服务端执行（不信任客户端）？

#### 5.2 输入验证
- [ ] 所有用户输入是否经过验证（Zod / class-validator）？
- [ ] 文件上传是否限制类型和大小？
- [ ] SQL 注入防护：是否使用 ORM 参数化查询？
- [ ] XSS 防护：前端是否转义用户输入内容？

#### 5.3 输出安全
- [ ] 错误响应是否不暴露内部堆栈或数据库细节？
- [ ] 敏感数据（密码、token）是否不出现在日志中？

#### 5.4 基础设施
- [ ] 速率限制是否配置（认证端点 5 req/min，普通端点 60 req/min）？
- [ ] CORS 配置是否限制来源？
- [ ] Cookie 是否设置 HttpOnly / Secure / SameSite？
- [ ] 是否启用 Helmet 安全头？

#### 5.5 依赖安全
- [ ] 新增依赖是否有已知漏洞（检查 npm audit）？
- [ ] 依赖版本是否锁定？

## 六、Spec 对齐审查（Spec Alignment Review）

### 触发时机
- 编码完成后，联调前
- 用户说"对照 spec 检查一下"

### 检查方法

1. 读取 `docs/issues/{dir}/specs/` 下的三层规格
2. 逐条检查验收标准
3. 输出对齐表格

**输出模板：**

```markdown
## Spec 对齐检查

### 功能规格对齐

| 用户故事 | 状态 | 实现位置    | 备注 |
|----------|------|-------------|------|
| {故事 1} | ✅/❌  | {文件:行号} |      |

### 行为规格对齐

| 交互状态 | 状态 | 实现位置    | 备注 |
|----------|------|-------------|------|
| loading  | ✅/❌  | {文件:行号} |      |
| empty    | ✅/❌  | {文件:行号} |      |
| error    | ✅/❌  | {文件:行号} |      |

### API 规格对齐

| 端点          | 状态 | 实现位置    | 备注 |
|---------------|------|-------------|------|
| POST /api/... | ✅/❌  | {文件:行号} |      |

### 错误码对齐

| 错误码 | 场景   | 状态 | 实现位置    |
|--------|--------|------|-------------|
| 400    | {场景} | ✅/❌  | {文件:行号} |
```

## 七、关闭前验收（Pre-Closure Acceptance）

### 触发时机
- issue 标记为 `in-review` 状态时
- 用户说"可以关闭了吗"、"验收一下"

### 验收清单

#### 7.1 代码验收
- [ ] 所有 Critical 和 Major 问题已修复
- [ ] 代码审查记录已保存到 `docs/reviews/`

#### 7.2 测试验收（TDD 强制检查）
- [ ] **`.spec.ts` 测试文件存在且可运行**
- [ ] **测试覆盖正常路径和错误路径**
- [ ] **单元测试全部通过**（`npx vitest run`）
- [ ] **类型检查通过**（`pnpm type-check`）
- [ ] **如有端到端测试，全部通过**
- [ ] **测试是在实现之前编写的**（抽查 git log 或询问开发者）

#### 7.3 Spec 验收
- [ ] 所有验收标准已勾选 `[x]`
- [ ] 实现与 spec 一致（无偏离）

#### 7.4 联调整合（前后端协作功能）
- [ ] 前端已移除 Mock，对接真实 API
- [ ] 端到端流程可正常运行
- [ ] 交互状态按 behavior-spec 实现
- [ ] API 按 api-spec 返回正确错误码

**验收通过标准**：所有必选项通过，无 Critical 问题。

## 八、TDD 合规审查（TDD Compliance Review）

### 触发时机
- 编码完成后，代码审查前
- 用户说"测试够吗"、"TDD 合规吗"、"检查测试覆盖"

### 检查清单

#### 8.1 测试存在性
- [ ] 新增功能是否有对应 `.spec.ts` 测试文件？
- [ ] 测试文件是否放在 `tests/issues/{dir}/` 下？
- [ ] 测试用例名是否以 `AC-XX:` 开头？

#### 8.2 测试覆盖度
- [ ] 是否覆盖正常路径和错误路径？
- [ ] 是否覆盖 behavior-spec 中的所有交互状态（loading/empty/error/success/partial）？
- [ ] 是否覆盖 api-spec 中的所有错误码场景？
- [ ] 边界条件是否测试？

#### 8.3 测试先行
- [ ] 测试是否在实现代码之前编写？（抽查 git log）
- [ ] 每个 plan 任务是否以"编写失败测试"开始？

#### 8.4 测试可运行
- [ ] 测试是否可运行且通过？（`npx vitest run`）
- [ ] 是否使用真实数据库（集成测试不 mock）？

**测试红线**：
- 无 `.spec.ts` 文件 → 🔴 Critical
- 测试只有 happy path → 🟠 Major
- 测试在实现之后补写 → 🟠 Major（首次警告，重复违规升级 Critical）

## 九、Plan 审查（Plan Review）

### 触发时机
- 计划编写完成后，执行前
- 用户说"plan 有没有问题"、"审查一下计划"

### 检查清单

#### 8.1 完整性
- [ ] 是否覆盖了 issue 中的所有验收标准？
- [ ] 是否覆盖了 spec 中的所有交互状态/端点/错误码？

#### 8.2 可执行性
- [ ] 每个步骤是否 2~5 分钟可完成？
- [ ] 是否包含具体代码示例？
- [ ] 是否包含验证命令？
- [ ] 是否有占位符（"TODO"、"稍后实现"、"TBD"）？

#### 8.3 TDD 合规性（新增）
- [ ] **每个任务是否以"编写失败测试"开始？**
- [ ] **每个任务是否以"运行测试确认通过"结束？**
- [ ] **测试文件路径是否明确声明？**
- [ ] **测试是否覆盖 spec 中的所有交互状态？**

#### 8.4 依赖关系
- [ ] 前置依赖是否清晰标注？
- [ ] 阻塞 issue 是否已解决？

#### 8.5 文件影响范围
- [ ] 新增/修改的文件列表是否完整？
- [ ] 是否有意外影响的其他文件？

## 十、审查后处理

### 9.1 问题修复跟踪

1. 在对话中输出审查结果（按第 3 节格式）
2. 当场修复或记录待办
3. 修复后运行验证命令确认
4. 如有遗漏，补充审查

### 9.2 Issue 状态更新

审查完成后，更新 issue 状态：
- 通过 → `closed`
- 有问题需修复 → 保持 `in-review`，在对话中记录待修复项
- 不通过 → 回到 `in-progress`

## 十二、子代理驱动模式下的审查派发（融入自 requesting-code-review）

当使用 `subagent-driven-development` 执行开发时，审查通过子代理派发。

### 何时请求审查

**强制：**
- 子代理驱动开发中每个任务后
- 完成主要功能后
- 合并到 main 前

**可选但 valuable：**
- 卡住时（新视角）
- 重构前（基线检查）
- 修复复杂 bug 后

### 如何派发审查子代理

**1. 获取 git SHAs：**

```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 使用 Task 工具派发 code-reviewer 子代理：**

填写模板：
- `{WHAT_WAS_IMPLEMENTED}` — 你刚构建了什么
- `{PLAN_OR_REQUIREMENTS}` — 它应该做什么（引用 plan 或 spec）
- `{BASE_SHA}` — 起始 commit
- `{HEAD_SHA}` — 结束 commit
- `{DESCRIPTION}` — 简要总结

**3. 处理反馈：**

| 级别 | 处理方式 |
|------|----------|
| Critical | 立即修复后才能继续 |
| Important | 继续前修复 |
| Minor | 记录待后续处理 |
| reviewer 错了 | 用技术推理反驳，必要时引用测试/代码证明 |

### 审查顺序（强制）

子代理驱动模式下，审查必须按此顺序：

1. **Spec 对齐审查** — 代码是否符合 spec？
2. **代码质量审查** — 实现是否优雅？

**代码质量审查在 spec 对齐通过前禁止开始。**

### 审查循环

reviewer 发现问题 → implementer 修复 → reviewer **重新审查** → 通过或再修复

**禁止跳过重新审查。**

## 十三、接收审查反馈的原则（融入自 receiving-code-review）

### 核心原则

**技术严谨优于社交舒适。**

### 禁止的回应

**永远不要：**
- "你说得对！"（表演式认同）
- "好点子！" / " excellent feedback!"（过度社交化）
- "我这就改"（未经验证就实现）

**正确的回应：**
- 用技术语言重述需求
- 提出澄清问题
- 直接开始工作（行动 > 言语）

### 处理流程

```
收到审查反馈时：
1. 完整阅读，不立即反应
2. 用自己的话重述要求（或提问澄清）
3. 对照代码库验证技术正确性
4. 评估：对本项目技术可行？
5. 回应：技术确认或有理有据的反驳
6. 实现：一次一项，每项测试
```

### 模糊反馈处理

**有任何不清楚的反馈项时：**
- **停止** —— 不要开始实现
- **询问** —— 澄清不清楚的项

**为什么：** 项与项之间可能有关联。部分理解 = 错误实现。

### YAGNI 检查

审查者建议"完善地实现"某功能时：
1. grep 代码库确认实际使用情况
2. 未使用 → "此功能未被调用。删除它（YAGNI）？"
3. 已使用 → 按规范实现

### 何时反驳

- 建议破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 对本技术栈不正确
- 存在兼容性/遗留原因
- 与架构决策冲突

**如何反驳：**
- 使用技术推理，不是防御性态度
- 提出具体问题
- 引用工作正常的测试/代码
- 涉及架构时邀请人类参与

### 修复顺序

多项目反馈时：
1. 先澄清所有不清楚的项
2. 按此顺序实现：
   - 阻塞性问题（功能破坏、安全）
   - 简单修复（拼写、导入）
   - 复杂修复（重构、逻辑）
3. 每项单独测试
4. 验证无回归

## 十四、常见陷阱

| 陷阱                | 后果                      | 正确做法                             |
|---------------------|---------------------------|--------------------------------------|
| 只审查代码不读 spec | 遗漏交互状态或错误码      | 始终先读 spec 再审查                 |
| 忽略边界条件        | 线上出现空状态/错误态崩溃 | 对照 behavior-spec 检查所有状态      |
| 安全审查流于形式    | 漏洞上线                  | 对照 q-01-security-baseline 逐项检查 |
| 发现问题不分级      | 阻塞非阻塞问题混为一谈    | 使用 Critical/Major/Minor/Info 四级  |
| 验收不检查联调      | 前后端接口不匹配          | 必须端到端验证后再关闭               |
| **不检查测试代码**  | TDD 流于形式              | 必须检查 `.spec.ts` 存在性和覆盖度   |
| **不验证测试先行**  | 事后补测试                | 抽查 git log 确认测试先于实现        |
